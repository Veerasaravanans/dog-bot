1. Project Overview & Objectives

What is DogBot?

Imagine a small robot car with a camera on top that can see its surroundings, understand what's in front of it (people, chairs, walls), decide which way to go, and drive itself without hitting anything — all while you watch the live video on your laptop browser. That's DogBot.

You can control it manually like a remote-control car, or switch it to autonomous mode where it drives itself using AI. It can even be controlled over the internet from a different city using cloud messaging.

DogBot Recon System is an autonomous reconnaissance robot that combines:

  • ESP32-CAM microcontroller for live video streaming & motor control
  • Computer Vision (OpenCV) for real-time obstacle detection & distance estimation
  • Deep Learning (YOLOv8n-seg) for object detection & instance segmentation
  • Autonomous Path Planning (5-stage pipeline with Kalman filtering) for safe navigation
  • AI-powered Decision Making via VIO Cloud LLM (Claude) as a fallback advisor
  • MQTT over TLS for remote internet-based control
  • Web Dashboard with real-time video, telemetry, and manual controls

The robot can operate in MANUAL mode (human-controlled via dashboard) or SEMI_AUTO mode (AI-driven with human override capability).

Project Goals

  1. Build a low-cost reconnaissance robot using commodity hardware (ESP32-CAM + L293D motor driver)
  2. Implement multi-sensor fusion for reliable obstacle detection (CV + ML + optical flow)
  3. Achieve real-time autonomous navigation with a 5-stage path planning pipeline (~2ms latency)
  4. Enable remote operation over the internet via MQTT cloud broker
  5. Provide a tactical web-based dashboard for monitoring and control

3. System Architecture

What is the "Service Orchestration Pattern"?

Think of a restaurant kitchen. There's a head chef (FrameManager) who coordinates all the station chefs: one handles the camera feed (ESP32Client), one analyzes the image (CVPipeline), one identifies objects using AI (MLDetector), one decides where to go (AIDecisionEngine), and one plans the exact route (PathPlannerEngine). Each "chef" does their job independently, but the head chef makes sure they work together in the right order and at the right time.

Why this pattern? It keeps code organized — each service is in its own file, can be tested independently, and can be started/stopped without affecting others. If one service fails, the rest keep running.

4. Data Flow Pipeline

What is a "Data Flow Pipeline"?

A pipeline is like an assembly line in a factory. A raw camera image enters at one end, passes through multiple processing stations (each adding something), and comes out the other end as a fully analyzed frame with obstacle positions, navigation decisions, and motor commands. This happens 15 times per second (15 FPS).

How the System Actually Works (Step by Step)

Step 1: You Start the Server

You type python run.py on your laptop. Here's what happens behind the scenes:

  1. On Windows, a special event loop (SelectorEventLoop) is created for MQTT compatibility
  2. FastAPI server starts on http://localhost:8000
  3. The lifespan manager kicks in and starts all services in order:
    • MQTTBridge connects to HiveMQ Cloud broker (for remote control)
    • ESP32Client starts reading the video stream from ESP32-CAM (or falls back to your laptop webcam)
    • MLDetector loads the YOLOv8 AI model into memory (~6.7 MB)
    • AIDecisionEngine initializes the path planner and decision logic
    • FrameManager starts the main processing loop at 15 FPS

Step 2: Video Frames Are Captured

The ESP32-CAM continuously captures JPEG photos at ~30 FPS and sends them over WiFi as an MJPEG stream. The ESP32Client on your laptop receives these frames by reading the HTTP stream and decoding each JPEG image. If the ESP32 is unreachable (powered off, different network), it automatically switches to your laptop's built-in webcam.

Step 3: Each Frame Is Processed (15 times per second)

The FrameManager grabs the latest frame and sends it through the processing pipeline:

  1. CVPipeline analyzes the frame using 4 methods (background subtraction, edges, gradients, optical flow) to find obstacles, estimate their distances, and classify lanes as clear/caution/blocked
  2. MLDetector runs YOLOv8 on the frame to identify specific objects (person, chair, etc.) with segmentation masks and tracking IDs
  3. AIDecisionEngine receives all the obstacle and detection data, runs the path planner, and decides: FORWARD, LEFT, RIGHT, or STOP
  4. The frame is annotated with colorful overlays showing obstacles, trajectories, telemetry, and the chosen path

Step 4: Path Planner Finds the Best Route (~2ms)

Inside the AIDecisionEngine, the 5-stage path planner runs:

  1. Build a bird's-eye-view occupancy grid (60×80 cells) from all detected obstacles
  2. Smooth obstacle positions using Kalman filters and predict where they'll be in the future
  3. Generate a cost map (red = dangerous, green = safe)
  4. Evaluate 45 possible curved paths and score each one for safety, progress, and smoothness
  5. Pick the best path and smooth the steering to prevent jerky movement

Step 5: Motor Commands Are Sent

The decided direction is sent to the ESP32's motors:

  • If MQTT is connected: Published to the dogbot/cmd/motor topic → ESP32 receives it instantly via cloud broker
  • If MQTT is down: Falls back to direct HTTP request to the ESP32's IP address (local network only)

The robot uses pulse-based control: move for 500ms → stop → analyze new frame → decide again. This prevents the robot from driving "blind" between decisions.

Step 6: Dashboard Shows Everything Live

Open http://localhost:8000 in your browser. Two WebSocket connections are established:

  • Video WebSocket receives 15 annotated frames per second — you see the live feed with colored obstacles, trajectory paths, and zone indicators
  • Control WebSocket lets you send manual motor commands (buttons/keyboard) and receives AI decisions, detection alerts, and telemetry

You can switch between MANUAL mode (you drive) and SEMI_AUTO mode (AI drives, you can override anytime).

5. ESP32-CAM Firmware (Embedded Layer)

File: firmware/esp32cam_dogbot/esp32cam_dogbot.ino

What is ESP32-CAM?

The ESP32-CAM is a tiny, low-cost microcontroller board (about the size of a matchbox) that has a camera, WiFi, and a processor all built into one chip. Think of it as a very basic smartphone without a screen — it can take pictures, connect to WiFi, and run simple programs.

Where is it used? It's the robot's physical hardware — mounted on the robot car. It captures live video and sends it to the Python backend over WiFi. It also receives motor commands and controls the wheels.

Why ESP32-CAM?

  • Cheap — Costs around $5-10, making it ideal for a low-cost robot
  • All-in-one — Camera + WiFi + processor in one board, no extra modules needed
  • Arduino-compatible — Easy to program using the Arduino IDE (beginner-friendly)
  • Dual-core — Can run camera streaming and motor control simultaneously without lag

What is Firmware?

Firmware is the software that runs directly on the hardware (ESP32 chip). Unlike Python code that runs on your laptop, firmware is written in C++ and uploaded (flashed) directly onto the microcontroller. Once uploaded, it runs automatically whenever the ESP32 is powered on — no computer needed.

What is MJPEG Streaming?

MJPEG stands for Motion JPEG. It's the simplest way to stream video: the camera takes a JPEG photo, sends it, takes another photo, sends it, and so on — like a flipbook animation. Each "page" is a separate JPEG image. At 30 photos per second, it looks like smooth video. It's not as efficient as modern video codecs (like H.264), but it's much simpler to implement on a microcontroller.

What is FreeRTOS?

FreeRTOS is a mini operating system for microcontrollers (like a very lightweight version of Windows/Linux). It lets the ESP32 run multiple tasks at the same time. In our case, one task handles the camera stream (on CPU Core 0), and another handles motor commands and MQTT (on CPU Core 1). Without FreeRTOS, the ESP32 could only do one thing at a time.

Motor Control (L293D + PWM)

  • PWM via LEDC: 1 kHz frequency, 8-bit resolution (0-255 speed range)
  • Tank steering: Left turn = Motor A backward + Motor B forward; Right turn = reverse
  • Motor trim: MOTOR_TRIM = 15 compensates hardware drift (reduces right motor speed by 15)
  • Lazy PWM initialization: PWM channels configured on first motor use to avoid ESP32 WiFi conflicts
  • Safety failsafe: Auto-stop after 800ms of no commands

  • Subscribes to: dogbot/cmd/motor — receives JSON {"dir":"forward","speed":200}
  • Publishes to: dogbot/status (every 5s) and dogbot/heartbeat (every 3s)
  • Auto-reconnect every 5 seconds on disconnect
  • Certificate validation skipped with setInsecure() (development mode)

6. FastAPI Backend (Server Layer)

Files: backend/main.py, backend/config.py

What is an API?

An API (Application Programming Interface) is a set of rules that allows different software programs to talk to each other. Think of it like a menu in a restaurant — you (the client/browser) look at the menu (API), place an order (send a request), and the kitchen (server) prepares your food (processes the request) and delivers it back to you (sends a response).

For example, when the dashboard wants the robot's current status, it sends a request to GET /api/health, and the backend responds with health data in JSON format.

What is FastAPI?

FastAPI is a modern Python web framework for building APIs. It's called "Fast" for two reasons:

  • Fast to run — It uses asynchronous programming (async/await), so it can handle many requests at the same time without getting stuck waiting
  • Fast to code — It automatically generates API documentation, validates data types, and has clean syntax

Why FastAPI over Flask or Django?

  • Native async/await support — Flask and Django are synchronous by default, meaning they handle one request at a time. FastAPI handles many simultaneously, which is critical for our robot (video stream + motor commands + MQTT + WebSocket all happening at once)
  • Built-in WebSocket support — We need real-time video streaming; FastAPI supports WebSocket natively
  • Pydantic integration — Automatic data validation (if the dashboard sends an invalid motor command, FastAPI catches it before it reaches the robot)
  • Automatic API docs — Visit /docs to see an interactive Swagger UI for testing all endpoints

What is Async/Await?

Normally, code runs one line at a time and waits for each operation to finish before moving to the next. This is called synchronous execution. But what if you're waiting for the ESP32 to send a video frame (takes 30ms) — should everything else freeze?

Async/await solves this. It lets your code say "start this task, and while waiting for the result, go do something else." It's like a chef in a kitchen: instead of staring at the rice cooker until it's done, the chef starts chopping vegetables. When the rice is ready, the chef comes back to it.

In our system, async enables: reading the video stream, handling WebSocket clients, running MQTT, and making LLM API calls — all concurrently in a single thread without blocking each other.

# Synchronous (SLOW - everything waits):
frame = get_frame()         # Wait 30ms... doing nothing
obstacles = detect(frame)   # Wait 50ms... doing nothing
send_to_dashboard(frame)    # Wait 10ms... doing nothing

# Asynchronous (FAST - work while waiting):
frame = await get_frame()         # While waiting, handle other WebSocket clients
obstacles = await detect(frame)   # While waiting, process MQTT messages
await send_to_dashboard(frame)    # While waiting, read next frame

What is a Lifespan Manager?

A lifespan manager is code that runs when the server starts up and shuts down. Think of it like opening and closing a restaurant:

  • Opening (startup): Turn on lights, heat up ovens, prep ingredients → Start MQTT connection, connect to camera, load AI model
  • Closing (shutdown): Turn off ovens, clean up, lock doors → Stop all services, close connections, free resources

Services must start in the right order (can't process video before the camera is connected) and stop in reverse order (can't close the camera before the video processor stops using it).

Application Lifecycle (Lifespan Pattern)

FastAPI uses @asynccontextmanager for startup/shutdown lifecycle:

@asynccontextmanager
async def lifespan(app: FastAPI):
    # STARTUP: Wire services → Start in dependency order
    await mqtt_bridge.start()    # 1. MQTT connection
    await esp32.start()          # 2. Video stream capture
    await ml_detector.start()    # 3. Load YOLOv8 model
    await ai_engine.start()      # 4. AI decision loop
    await frame_mgr.start()      # 5. Main processing loop

    yield  # App is running

    # SHUTDOWN: Stop in reverse order
    await frame_mgr.stop()
    await ai_engine.stop()
    await esp32.stop()
    await mqtt_bridge.stop()

Asyncio Service Lifecycle Pattern (used by ALL services):

async def start(self):
    self._running = True
    self._task = asyncio.create_task(self._loop())

async def stop(self):
    self._running = False
    if self._task:
        self._task.cancel()
        try: await self._task
        except asyncio.CancelledError: pass

Windows Event Loop Fix (Critical)

run.py solves a Windows-specific issue where the default ProactorEventLoop doesn't support add_reader()/add_writer() needed by aiomqtt/paho-mqtt:

# run.py — Windows MQTT fix
if sys.platform == 'win32':
    loop = asyncio.SelectorEventLoop()  # Supports add_reader/add_writer
    asyncio.set_event_loop(loop)
    loop.run_until_complete(main())      # Run uvicorn.Server in this loop
else:
    asyncio.run(main())                  # Linux/Mac — default works fine

Why: uvicorn.run() creates its own event loop, but we need to control which loop type is used. By creating a SelectorEventLoop explicitly and using uvicorn.Server (not uvicorn.run), all async code runs in the correct loop.

7. Computer Vision Pipeline

File: backend/services/cv_pipeline.py (~1100 lines)

What is Computer Vision?

Computer Vision (CV) is the field of teaching computers to "see" and understand images. When you look at a photo, your brain instantly recognizes objects, estimates distances, and detects motion. Computer Vision tries to do the same thing using algorithms and math.

What is OpenCV?

OpenCV (Open Source Computer Vision Library) is the world's most popular computer vision library with 2,500+ algorithms. It provides ready-made functions for image processing — blurring, edge detection, object tracking, etc. Instead of writing complex image processing math from scratch, we call OpenCV functions. It's free, fast (written in C++ with Python bindings), and used by companies like Google, Tesla, and NASA.

Why use multiple detection methods?

No single detection method is perfect. Background subtraction fails if the robot is moving. Edge detection can be confused by textures. Optical flow only sees moving objects. By combining 4 different methods, we get reliable detection even when individual methods fail. This is called multi-sensor fusion — like having 4 people watching for obstacles, each looking for different things. If 3 out of 4 agree something is there, it probably is.

Advantages of this approach:

  • More reliable than any single method alone
  • No AI model needed — runs on pure math/algorithms (fast, no GPU required)
  • Works in real-time on every frame (not frame-skipped like ML detection)
  • Detects both static and moving obstacles

Multi-Sensor Fusion Approach

The CV pipeline combines 4 detection methods for robust obstacle detection:

1. Background Subtraction (MOG2)

In simple terms: The camera watches the scene for a while and learns what the "background" looks like (walls, floor, furniture). Anything new that appears (a person walking in) is detected as "foreground" — a potential obstacle.

  • Algorithm: cv2.createBackgroundSubtractorMOG2(history=300, varThreshold=40)
  • Learns a statistical model of the background over 300 frames
  • Detects foreground objects as deviations from the learned background
  • Shadow detection enabled for handling partial occlusions

Advantage: Great at detecting ANY new object in the scene, regardless of shape or color. Limitation: Needs the camera to be stationary; struggles when the robot itself is moving (everything shifts).

2. Canny Edge Detection

In simple terms: Finds the sharp boundaries (edges) of objects in the image. Where the color changes suddenly (e.g., a dark chair against a light floor), Canny draws a line. These edge lines reveal obstacle shapes.

  • Parameters: low=50, high=150 (configurable via .env)
  • Pre-processing: Gaussian blur with kernel size 5
  • Detects object boundaries for contour-based obstacle identification

Advantage: Works even when the robot is moving, doesn't need a "learned" background. Simple and very fast.

3. Sobel Gradient Magnitude

In simple terms: Similar to Canny but measures the strength of edges rather than just detecting them. It calculates how quickly the image brightness changes in each direction (horizontal and vertical), which helps find solid object boundaries.

  • Computes grad_x and grad_y using Sobel operators (ksize=3)
  • Magnitude: √(grad_x² + grad_y²)
  • Threshold at 30 for binary mask — texture-agnostic edge detection
  • Also used for thin-edge map: filters for vertical structures (poles, legs, wires)

Advantage: Specifically good at detecting thin vertical structures like table legs, poles, and wires that other methods might miss.

4. Dense Optical Flow (Farneback)

In simple terms: Compares two consecutive frames and calculates how each pixel has moved. If a group of pixels is moving together (like a person walking), that's a moving obstacle. It's like spotting someone in a crowd by noticing they're the one moving.

  • Algorithm: cv2.calcOpticalFlowFarneback()
  • Parameters: pyr_scale=0.5, levels=3, winsize=15, iterations=3
  • Detects moving obstacles by computing pixel motion between consecutive frames
  • Flow threshold: 2.0 pixels/frame for motion significance
  • Computed every 2 frames for performance

Advantage: The ONLY method that detects motion. A person walking toward the robot is a bigger threat than a stationary chair, and optical flow catches this.

Perspective-Based Distance Estimation

Uses inverse perspective mapping to estimate real-world distance from pixel position:

distance_m = DIST_SCALE / (pixel_y - vanishing_point_y)
# DIST_SCALE = 80.0 (calibration constant)
# vanishing_point_y = 0.30 × frame_height = 144 pixels (for 480p)

Vanishing Point Auto-Detection:

Every 30 frames, the system uses Probabilistic Hough Transform to find line intersections and estimate the vanishing point for improved perspective calibration.

Lane Classification & Free-Path Analysis

  • Frame divided into 3 lanes: LEFT (0-33%), CENTER (33-67%), RIGHT (67-100%)
  • Each lane classified as: clear, caution, or blocked
  • Free path recommendation: safest lane for navigation
  • Nearest obstacle distance tracked per lane

Centroid-Based Multi-Object Tracking

  • Hungarian-style greedy matching: sorted by Euclidean distance between centroids
  • Max association distance: 80 pixels
  • Track age management: tracks die after 8 frames without match
  • Velocity estimation: computed from previous centroid position
  • Time-to-Collision (TTC) calculated from closing speed

Auxiliary Maps for Path Planner

  • Thin Edge Map: Detects thin vertical structures (desk legs, poles, wires) that standard contour detection misses
  • Floor Color Model: Adaptive HSV model using Exponential Moving Average (EMA, α=0.05) to learn the floor color and flag non-floor regions as potential obstacles

8. ML Object Detection (YOLOv8)

File: backend/services/ml_detector.py

What is Object Detection?

Object detection is an AI task where a computer looks at an image and identifies what objects are in it and where they are. For example: "There is a person at coordinates (100, 200) and a chair at coordinates (400, 300)." This is done by a trained neural network (deep learning model) that has "seen" millions of images during training.

What is YOLO?

YOLO (You Only Look Once) is a family of object detection models. Unlike older methods that scan an image multiple times in small regions, YOLO processes the entire image in one pass (one "look"), making it extremely fast. YOLOv8 is the latest version from Ultralytics with improved accuracy and speed.

What is Instance Segmentation?

Regular object detection draws a rectangle (bounding box) around each object. Instance segmentation goes further — it identifies the exact pixel-level outline of each object. This gives us the precise shape of obstacles (not just a box) for more accurate path planning.

Example: A bounding box around a person includes empty space on both sides. A segmentation mask traces exactly around the person's body, so the path planner knows exactly where the obstacle really is.

Why YOLOv8n-seg (Nano Segmentation)?

  • Nano = smallest and fastest variant (~6.7 MB). It runs on CPU without a GPU, which is important for a laptop-based robot controller
  • Segmentation = pixel-level masks give more precise obstacle shapes than bounding boxes
  • 80 classes = pre-trained on COCO dataset, recognizes person, chair, bottle, dog, car, etc. — no custom training needed
  • Built-in tracking = ByteTrack tracker maintains persistent IDs across frames (person #1 stays #1 even if they move)

Advantage over CV Pipeline: The CV pipeline detects "something is there" but doesn't know what it is. YOLOv8 tells you it's a "person" with 85% confidence. This enables smarter decisions (e.g., slow down for a person, ignore a shadow).

9. Path Planning Engine (5-Stage Pipeline)

Files: backend/services/path_planner/ (engine.py, occupancy_grid.py, obstacle_estimator.py, cost_map.py, trajectory_evaluator.py, schemas.py)

What is Path Planning?

Path planning is how the robot decides which way to go to reach its goal without hitting anything. Imagine you're walking through a crowded room — your brain constantly looks at obstacles around you, predicts where moving people are heading, evaluates multiple possible paths, and picks the safest one. The path planner does exactly this, but using math and algorithms, 15 times per second.

What is a Kalman Filter? (Detailed Explanation)

The Kalman filter is one of the most important algorithms in robotics. Here's a simple way to understand it:

The Problem: Our camera detects an obstacle at position X. But camera measurements are noisy (imprecise) — the detected position jumps around slightly each frame. If the path planner reacts to these noisy measurements directly, the robot would zigzag erratically.

The Solution: The Kalman filter combines two sources of information:

  1. Prediction (what we EXPECT): "The obstacle was at position 2.0m last frame, moving at 0.5 m/s. So this frame, it should be at approximately 2.0 + 0.5 × 0.066 = 2.033m."
  2. Measurement (what we SEE): "The camera says the obstacle is at 2.05m."

The filter blends these two: prediction says 2.033m, measurement says 2.05m. The filter calculates a weighted average based on how much it trusts each source (using the Kalman Gain). If measurements have been accurate recently, it trusts the measurement more. If measurements are noisy, it trusts the prediction more.

Real-world analogy: Imagine tracking a friend's location using GPS (noisy, jumps around) AND your knowledge of where they were walking (they were heading north at walking speed). The Kalman filter combines both to give you a smooth, accurate estimate of their actual position.

Two Steps Every Frame:

  • PREDICT: Use the motion model to estimate where the obstacle is NOW (before looking at the camera)
  • UPDATE: Look at the camera measurement, compute Kalman Gain, and correct the prediction

Why it's powerful in our system:

  • Smooths noisy position measurements → robot doesn't zigzag
  • Estimates obstacle velocity (speed and direction of movement)
  • Predicts future positions — "this person will be HERE in 0.5 seconds" → robot can avoid the predicted position proactively

What is an Occupancy Grid?

Imagine looking at the room from above (bird's-eye view) and dividing the floor into a grid of small squares (5cm × 5cm each). Each square is colored: white = free space, black = obstacle. This grid is the occupancy grid — a simple 2D map of what's around the robot. The path planner uses this map to find clear paths.

Our grid: 60 columns × 80 rows = 3 meters wide × 4 meters deep, with 5cm resolution.

What is DWA (Dynamic Window Approach)?

DWA is a path planning method that works by trying many possible paths and picking the best one. Think of it like choosing a lane on a highway:

  1. Generate 45 possible curved paths (different steering angles × different speeds)
  2. For each path, check: "Will I hit something? How close will I get to obstacles? Am I going forward?"
  3. Score each path using multiple criteria (safety, progress, smoothness)
  4. Pick the path with the best score

Advantage: Very fast (~2ms per frame) because it evaluates a fixed number of candidates. It accounts for the robot's physical limits (minimum turn radius, maximum speed).

Overview

The path planner is the core navigation brain. It runs every frame (~2ms latency) and produces continuous steering (-1 to +1) and speed (0 to 1) commands.

    dist_m = DIST_SCALE / (py - vanish_y)  # Forward distance
    frac = (px - FRAME_W/2) / (FRAME_W/2)  # -1 to +1 lateral
    scale = min(dist_m, depth_m) / depth_m
    lateral_m = frac * (width_m/2) * max(scale, 0.15)
    return (lateral_m, dist_m)

Stage 2: Obstacle State Estimation (Kalman Filter)

File: obstacle_estimator.py

  • Filter type: Constant-velocity Kalman filter (4 states: x, y, vx, vy)
  • Per-obstacle tracking: Each tracked obstacle ID gets its own Kalman filter instance
  • State vector: [x_position, y_position, x_velocity, y_velocity] in world metres
  • Measurement: (x, y) position from pixel-to-world conversion
  • Prediction step: state = F @ state where F has dt in velocity coupling positions
  • Update step: Standard Kalman gain: K = P @ H.T @ inv(H @ P @ H.T + R)
  • Stale removal: Tracks removed after 10 frames unseen
  • Confidence decay: Unseen tracks lose 10% confidence per frame

Kalman Filter Matrices:

F = [[1, 0, dt, 0],    # State transition (constant velocity)
     [0, 1, 0, dt],
     [0, 0, 1,  0],
     [0, 0, 0,  1]]

H = [[1, 0, 0, 0],    # Observation (measure position only)
     [0, 1, 0, 0]]

Q = diag([0.01, 0.01, 0.1, 0.1])   # Process noise
R = diag([0.05, 0.05])               # Measurement noise

Stage 5: Temporal Smoothing

  • Exponential Moving Average (EMA): α = 2/(5+1) = 0.33
  • smoothed = α × raw + (1-α) × previous
  • Prevents jerky steering from frame-to-frame noise
  • STOP bypasses smoothing (safety override)
  • Direction hold: 300ms minimum before allowing direction change (prevents oscillation)

  • Direction chosen by comparing left vs. right average cost map values
  • Recovery parameters: speed=0.5, steering=0.8

10. AI Decision Engine & LLM Integration

File: backend/services/ai_decision.py

What is the AI Decision Engine?

The AI Decision Engine is the "brain" that makes the final call on where the robot should go. It takes all the information from the CV pipeline, ML detector, and path planner, and decides: "Go forward", "Turn left", "Turn right", or "Stop".

What is an LLM (Large Language Model)?

An LLM is an AI model trained on massive amounts of text data that can understand and generate human language. ChatGPT, Claude, and Gemini are all LLMs. In our project, we use Claude (via VIO Cloud) as a backup brain — when the local path planner isn't confident about what to do, it asks the LLM for a "second opinion".

Why a Two-Tier Decision System?

Think of it like driving with a co-pilot:

  • Tier 1 (Local Path Planner): The main driver. Makes decisions every frame (~2ms). No internet needed. Handles 95% of situations.
  • Tier 2 (Cloud LLM): The co-pilot. Only consulted when the driver isn't sure (confidence < 40%). Takes 2-8 seconds (network call). Provides a "second opinion" with reasoning.

Advantage: The robot never freezes waiting for the cloud — the local planner always has an answer. The LLM only improves decisions in tricky situations. If the internet goes down, the robot keeps navigating with Tier 1 alone.

Decision Architecture

Two-tier decision making:

  1. Primary (Fast): Local PathPlannerEngine — runs every frame, ~2ms, no network calls
  2. Fallback (Slow): VIO Cloud LLM (Claude) — only when planner confidence < 0.4, runs in background, cached 10 seconds

Multi-Frame Direction Voting

Prevents single-frame noise from causing wrong turns:

  • Vote window: 5 frames (deque with maxlen=5)
  • Majority threshold: 60% consensus required
  • STOP override: Always immediate (safety), no voting needed
  • No consensus: Holds previous direction for stability
  • Generation check: Only votes on NEW planner outputs (prevents duplicate votes when decision loop is faster than frame pipeline)

Pulse-Based Motor Control

Instead of continuous commands, the robot uses a pulse pattern:

  1. Send motor command (e.g., FORWARD)
  2. Run for pulse duration (500ms)
  3. Auto-stop → wait for fresh frame analysis
  4. Decide again based on new scene data

Why: Ensures the robot analyzes the scene between every movement pulse, preventing it from driving blind.

What is MQTT?

MQTT (Message Queuing Telemetry Transport) is a lightweight messaging protocol designed for IoT (Internet of Things) devices. Think of it like a WhatsApp group:

  • There's a central broker (like the WhatsApp server) — in our case, HiveMQ Cloud
  • Devices subscribe to topics (like joining a group) to receive messages
  • Devices publish messages to topics (like sending a message to the group)
  • Any device subscribed to a topic gets the message instantly

Example in our system: The Python backend publishes {"dir":"forward","speed":200} to the topic dogbot/cmd/motor. The ESP32 is subscribed to this topic, so it receives the command and moves the motors forward.

12. WebSocket Communication

What is a WebSocket?

To understand WebSocket, first understand how normal HTTP works:

  • HTTP (normal web requests) is like sending letters: you send a request, wait for a response, and the conversation ends. If you want new data, you send another letter. This is fine for loading a webpage, but terrible for live video — you'd have to send 15 requests per second!
  • WebSocket is like a phone call: once the connection is established, both sides can talk freely at any time, without hanging up and calling again. The connection stays open until one side hangs up.

Where is it used in our project?

  • Video stream (/ws/video): The server pushes 15 frames per second to the browser. With HTTP, the browser would need to ask 15 times per second. With WebSocket, the server just keeps sending.
  • Control channel (/ws/control): The browser sends motor commands, and the server sends back AI decisions and alerts — both directions, anytime.

Why WebSocket over HTTP polling?

  • Lower latency — No overhead of opening/closing connections for each frame
  • Real-time — Server pushes data instantly instead of waiting for client to ask
  • Less bandwidth — No repeated HTTP headers for every frame
  • Bidirectional — Client and server can send data at the same time

What is Backpressure?

Imagine a water pipe: if water flows in faster than it flows out, the pipe bursts. Similarly, if the server sends frames faster than the browser can display them, frames pile up in memory and cause increasing lag. Backpressure solves this by setting a limit (queue of 2 frames) — if the client is slow, the oldest frame is dropped so the client always sees the latest frame, not a delayed one.

Video WebSocket (/ws/video)

  • Pattern: Pub/Sub with backpressure
  • Clients subscribe via frame_manager.subscribe() → get a asyncio.Queue(maxsize=2)
  • FrameManager broadcasts JSON FrameResult to all subscriber queues
  • Backpressure: If queue is full, oldest frame is dropped (prevents lag accumulation)
  • Data: Base64-encoded JPEG frame + telemetry + detections + obstacles

Control WebSocket (/ws/control)

  • Bidirectional: Client sends commands, server broadcasts decisions
  • Inbound messages:
    • {"type":"motor","direction":"forward","speed":200} — Manual motor command
    • {"type":"mode","value":"semi_auto"} — Switch control mode
  • Outbound messages:
    • {"type":"ai_decision",...} — AI navigation decision
    • {"type":"alert",...} — High-confidence detection alert
    • {"type":"motor_ack",...} — Motor command acknowledgement
    • {"type":"mode_change",...} — Mode switch confirmation

16. Key CS/Robotics Concepts Used

This section is a quick reference of all the technical concepts used in the project. Each concept includes a brief "in simple terms" explanation.

Computer Vision

  • Background Subtraction (MOG2): Learns what the scene normally looks like and flags anything new as an obstacle. Technical: Gaussian Mixture Model separating foreground from background.
  • Canny Edge Detection: Finds sharp boundaries between objects by detecting where brightness changes suddenly. Technical: Gaussian blur → Sobel gradient → non-maximum suppression → hysteresis thresholding.
  • Sobel Operators: Measures how quickly image brightness changes, used to detect edges and textures. Technical: First-order derivative filters for image gradients in x/y.
  • Dense Optical Flow (Farneback): Detects motion by comparing two consecutive frames pixel by pixel. Technical: Polynomial expansion method for computing motion field.
  • Hough Transform: Finds straight lines in an image, used to estimate where the floor meets the walls (vanishing point). Technical: Probabilistic line detection.
  • Morphological Operations: Cleans up noisy images by removing small dots (opening) and filling small gaps (closing). Technical: Uses elliptical/rectangular kernels.
  • Perspective Projection: Converts pixel positions in the camera image to real-world distances in meters. Technical: Inverse perspective mapping from pixel space to metric coordinates.
  • MJPEG Protocol: Streams video as a series of JPEG images sent one after another. Technical: Multipart HTTP with boundary markers.

Machine Learning

  • YOLOv8 (You Only Look Once): An AI model that identifies objects in a photo in a single pass, making it very fast. Technical: Single-shot real-time object detector; nano variant for CPU efficiency.
  • Instance Segmentation: Traces the exact outline of each object (not just a rectangle around it). Technical: Per-object pixel masks.
  • ByteTrack: Keeps track of the same object across multiple frames (e.g., "person #1 moved from here to there"). Technical: Multi-object tracking using high and low confidence detections.
  • Transfer Learning: Using an AI model pre-trained on millions of images instead of training from scratch. Technical: Pre-trained on COCO dataset (80 classes).
  • Non-Maximum Suppression (NMS): When the AI detects the same object multiple times, keep only the most confident detection. Technical: Removes overlapping bounding boxes.

Robotics & Control

  • Kalman Filter: Smooths noisy sensor readings and predicts where obstacles will be next. Like GPS smoothing your position. Technical: Optimal recursive estimator for linear dynamic systems.
  • Occupancy Grid: A bird's-eye-view map divided into small squares showing where obstacles are. Technical: Probabilistic spatial representation of environment.

  • Tank Steering: The robot turns by spinning its left and right wheels at different speeds or directions. Technical: Differential drive.
  • PWM Motor Control: Controls motor speed by rapidly switching power on/off. More ON time = faster speed. Technical: Pulse Width Modulation from digital GPIO via LEDC peripheral.

Q1: What is the overall architecture of the DogBot system?

Simple answer: The system has 3 parts that work together like a remote-controlled surveillance drone: (1) The robot itself (ESP32-CAM), (2) The laptop brain (Python backend), and (3) The control screen (web dashboard in the browser).

Detailed answer: DogBot follows a three-layer architecture: (1) Embedded layer — ESP32-CAM (a small microcontroller with a camera) runs Arduino C++ firmware. It captures live video as an MJPEG stream on port 81 and receives motor commands via HTTP (port 80) or MQTT (over the internet). (2) Backend layer — A Python FastAPI server running on your laptop acts as the brain. It receives the video feed, processes it with computer vision (OpenCV), detects objects with AI (YOLOv8), plans a safe path (5-stage pipeline with Kalman filtering), and sends motor commands back to the ESP32. (3) Frontend layer — A web dashboard (opened in Chrome/Firefox) that shows live video, detection overlays, telemetry, and manual controls. It connects to the backend via two WebSocket connections (one for video, one for control).

Q2: How does the obstacle detection work? What methods are used?

Simple answer: We use 4 different "eyes" to look for obstacles, each detecting different things. If multiple methods agree something is there, we're confident it's a real obstacle. On top of that, an AI model (YOLOv8) tells us what the object is (person, chair, etc.).

Detailed answer: We use multi-sensor fusion combining 4 methods: (1) Background Subtraction (MOG2) — the camera learns what the normal background looks like and flags anything new. Good for detecting any new object, but struggles when the robot moves. (2) Canny Edge Detection — finds sharp boundaries between objects by detecting where brightness changes. Fast and works while moving. (3) Sobel Gradient Magnitude — measures edge strength, especially good at detecting thin vertical structures like table legs and poles. (4) Dense Optical Flow (Farneback) — compares consecutive frames to detect moving objects. The only method that detects motion. Additionally, YOLOv8n-seg provides AI-based detection that identifies what each object actually is (person, chair, bottle). All results are combined in the occupancy grid, where each source contributes at different confidence levels.

Q3: What is a Kalman filter and how is it used here?

Simple answer: A Kalman filter is like GPS smoothing. Your GPS shows your location jumping around slightly (noisy data), but your phone smooths it out because it knows you're walking in a straight line. The Kalman filter does the same for obstacle positions — it smooths noisy camera measurements and predicts where obstacles will be next.

Detailed answer: A Kalman filter is an algorithm that combines two sources of information: (1) What we predict — based on where the obstacle was and how fast it was moving, we estimate where it should be now. (2) What we see — the camera's new measurement of the obstacle position. The filter blends these two using a "Kalman Gain" that automatically adjusts based on how trustworthy each source is. Technically, it maintains a state vector [x, y, vx, vy] (position + velocity) with an uncertainty matrix (covariance). Each frame has two steps: Predict (project state forward: x_new = x + vx×dt) and Update (correct prediction with new measurement). In our system, each tracked obstacle gets its own Kalman filter. This provides smooth positions and enables future prediction — "this person will be at position X in 0.5 seconds" — so the robot can dodge proactively.

Q4: Explain the path planning pipeline in detail.

Simple answer: Every frame (15 times per second), the robot builds a bird's-eye map of obstacles, smooths their positions, creates a "danger map", tries 45 possible paths, picks the safest one, and smooths the steering. All of this takes only ~2 milliseconds.

Detailed answer: Our path planner runs a 5-stage pipeline every frame (~2ms): Stage 1 — Occupancy Grid: Creates a 60×80 bird's-eye grid (3m×4m, 5cm resolution) by projecting CV obstacles, ML detections, edge maps, and floor masks using inverse perspective mapping. Stage 2 — Kalman Filtering: Per-obstacle constant-velocity Kalman filters smooth positions and estimate velocities. Stage 3 — Cost Map: Combines 5 weighted layers: occupancy (0.45), repulsive potential fields (0.25), predictive future positions (0.15), boundary cost (0.05), and attractive goal field (0.10). Stage 4 — Trajectory Evaluation (DWA): Generates 45 candidate circular arcs (15 steering × 3 speeds), samples 10 points along each, scores by obstacle/heading/smoothness/clearance/progress costs, rejects infeasible ones. Stage 5 — Temporal Smoothing: Exponential Moving Average (α=0.33) smooths steering and speed to prevent jitter.

Q5: What is YOLOv8 and why did you choose the nano-seg variant?

Simple answer: YOLO is an AI model that looks at a photo and instantly identifies every object in it. We use the smallest version ("nano", 6.7MB) with segmentation so it traces the exact shape of each object. It can run on a regular laptop CPU without needing an expensive GPU.

Detailed answer: YOLO (You Only Look Once) is a single-shot object detector — unlike older methods that scan an image piece by piece, YOLO processes the entire image in one pass, making it extremely fast. YOLOv8 is the latest version from Ultralytics. We chose YOLOv8n-seg (nano segmentation) because: (1) It's the smallest and fastest variant (~6.7MB), suitable for real-time CPU inference without a GPU. (2) The "-seg" variant provides instance segmentation masks (pixel-level outlines, not just rectangular boxes), giving the path planner precise obstacle shapes. (3) It's pre-trained on the COCO dataset (80 common objects: person, chair, dog, car, etc.), so no custom training is needed. (4) Built-in ByteTrack tracker maintains persistent object IDs across frames (person #1 stays #1 even as they move).

Q6: How does the MQTT communication work?

Simple answer: MQTT is like a WhatsApp group for devices. The ESP32 and the Python backend both join a cloud "group" (HiveMQ broker). When the backend sends a motor command, the ESP32 receives it instantly — even if they're in different cities. It works over the internet, unlike HTTP which only works on the local WiFi network.

Detailed answer: MQTT is a lightweight publish/subscribe messaging protocol designed for IoT devices. Both the ESP32 and Python backend connect to a HiveMQ Cloud broker over TLS (encrypted, port 8883). The backend publishes motor commands to the dogbot/cmd/motor topic with QoS 1 (guaranteed at-least-once delivery). The ESP32 subscribes to this topic and executes the commands. In the other direction, the ESP32 publishes its status (WiFi signal, motor state, uptime) to dogbot/status every 5 seconds and a heartbeat to dogbot/heartbeat every 3 seconds. Key advantage: This enables remote control over the internet — the operator can be in a completely different city. If MQTT is unavailable, the system falls back to direct HTTP (local network only).

Q7: What is the Dynamic Window Approach (DWA)?

DWA is a local path planning algorithm that evaluates trajectories within the robot's kinematic constraints. It works by: (1) Generating a set of candidate trajectories (we use 45 arcs: 15 steering angles × 3 speeds). (2) Each trajectory is a circular arc defined by steering and speed, sampled at 10 points over 1.5m look-ahead distance. (3) Trajectories are scored by multiple weighted costs: obstacle proximity, heading deviation, steering smoothness, clearance margin, and forward progress. (4) Infeasible trajectories (too close to obstacles or high cost) are rejected. (5) The best feasible trajectory is selected. This approach is efficient (~2ms) because it evaluates a finite set of candidates rather than searching a continuous space.

Q8: How does the system handle the ESP32 being unavailable?

The system has a graceful fallback chain: (1) The ESP32Client continuously tries to connect to the ESP32's MJPEG stream. (2) If the stream fails, it automatically activates the laptop webcam as a video source (640×480 @ 20 FPS). (3) The rest of the pipeline (CV, ML, path planning) works identically regardless of video source. (4) For motor commands, if MQTT is unavailable, it falls back to direct HTTP calls. (5) When ESP32 comes back online, the webcam is stopped and ESP32 stream resumes. The source field in telemetry shows "esp32", "webcam", or "none".

Q9: What is the multi-frame voting mechanism?

Simple answer: Instead of reacting to a single frame (which could be wrong due to camera noise), the robot looks at the last 5 decisions and goes with the majority. If 4 out of 5 say "go forward", it goes forward. This prevents random single-frame errors from causing the robot to zigzag. STOP is the exception — it's always immediate for safety.

Detailed answer: The AI decision engine maintains a deque (sliding window) of the last 5 direction decisions. A direction is only confirmed if it has ≥60% majority (at least 3 of 5 votes). If no direction achieves majority, the robot holds its previous direction for stability. STOP is always immediate (safety override — no voting needed). Additionally, the system only casts a new vote when a genuinely new planner output is available (tracked via generation counter), preventing the faster decision loop (15 Hz) from stuffing duplicate votes.

Q10: Why use FastAPI and async/await?

Simple answer: FastAPI is a Python web framework — think of it as the "waiter" that takes requests from the browser and delivers responses. We chose it because it can handle many things at once (video + control + MQTT) without slowing down, and it has built-in WebSocket support for live video.

Detailed answer: FastAPI was chosen because: (1) It's built on ASGI (Asynchronous Server Gateway Interface), enabling native async/await — this means the server can handle reading the video stream, processing MQTT messages, and responding to WebSocket clients all at the same time without blocking. (2) Built-in WebSocket support for real-time video streaming at 15 FPS. (3) Automatic interactive API documentation at /docs. (4) Pydantic integration for automatic data validation. Why not Flask/Django? Flask is synchronous (handles one request at a time) and doesn't support WebSocket natively. Django is too heavy for a lightweight robot controller. FastAPI gives us async, WebSocket, and validation in one lightweight package. CPU-intensive work (YOLOv8 inference) is offloaded to a thread pool via run_in_executor() so it doesn't freeze the async event loop.

Q11: How does the occupancy grid work?

The occupancy grid is a 2D bird's-eye-view representation of the environment. It's a 60×80 float array (3m × 4m, 5cm per cell) where each cell value represents the probability of an obstacle being present (0.0 = free, 1.0 = occupied). Building process: (1) Old grid values are decayed by 0.7× (temporal fading). (2) CV obstacle bounding box corners are projected from pixel space to world coordinates using inverse perspective mapping, then filled as polygons on the grid. (3) ML segmentation mask points are similarly projected. (4) Thin edges and floor anomalies are projected at low confidence (0.15-0.2). (5) A Gaussian inflation kernel (3×3) dilates obstacles for safety margin. The grid serves as input to the cost map generator.

Q12: What is the role of the LLM (Large Language Model) in this system?

The LLM serves as a fallback advisor, not the primary navigator. The primary navigation is handled by the local path planner (fast, ~2ms, no network). The LLM (Claude via VIO Cloud) is only queried when: (1) The local planner's confidence drops below 0.4, AND (2) The action is not STOP. The LLM query runs as a non-blocking background task — it never delays navigation. Results are cached for 10 seconds. If the cached LLM result has higher confidence than the local planner, it overrides the decision. This provides a "second opinion" for ambiguous scenarios while maintaining real-time responsiveness.

Q13: Explain the ESP32-CAM hardware and the motor driver circuit.

The ESP32-CAM (AI-Thinker module) has an ESP32-S chip, OV2640 camera, 4MB PSRAM, and WiFi. It runs at 240MHz with FreeRTOS dual-core. The camera captures VGA (640×480) JPEG frames using PSRAM double-buffering. The L293D motor driver controls two DC motors (left/right) for differential/tank steering. Each motor has 2 direction pins (IN1-IN4) and 1 PWM enable pin (ENA/ENB). PWM is generated using ESP32's LEDC (LED Control) peripheral at 1 kHz, 8-bit resolution (0-255 speed). GPIO pins: Motor A (left) = IN1:GPIO12, IN2:GPIO13, ENA:GPIO2; Motor B (right) = IN3:GPIO14, IN4:GPIO15, ENB:GPIO4. The MJPEG stream runs on Core 0 via FreeRTOS xTaskCreatePinnedToCore().

Q14: What is the difference between MANUAL and SEMI_AUTO modes?

In MANUAL mode: The human operator directly controls the robot via dashboard buttons or keyboard. The AI decision engine runs but does not send commands. All detection and telemetry still display. In SEMI_AUTO mode: The AI decision engine autonomously controls the robot using the path planner. It uses pulse-based control (move → stop → analyze → decide → move). Human input is still accepted and takes priority — when a manual command is received, the AI pauses for 5 seconds (configurable) before resuming. The mode is toggled via the dashboard and transmitted via WebSocket.

Q15: How does the cost map combine different factors?

The cost map is a weighted sum of 5 components: (1) Occupancy (0.45): Direct obstacle presence. (2) Repulsive field (0.25): Gaussian decay around each obstacle's current position (σ=4 cells), scaled by confidence. (3) Predictive (0.15): Same repulsive fields but at predicted future positions (0.3s, 0.6s, 1.0s ahead) using Kalman velocity estimates, with time-decay weighting. (4) Boundary (0.05): Pre-computed static cost that increases near grid edges. (5) Attractive (0.10): Pre-computed static cost that decreases toward the forward-center of the grid (goal direction). Final cost is clipped to [0, 1]. This creates a potential field that the trajectory evaluator navigates through.

Q16: What is WebSocket and how is it used differently from HTTP here?

Simple answer: WebSocket is like a phone call between the browser and the server — once connected, both sides can talk freely anytime. Normal HTTP is like sending letters (you send a request, get a response, done). WebSocket stays open so the server can push 15 video frames per second to your browser without the browser asking 15 times.

Detailed answer: WebSocket is a full-duplex communication protocol over a single TCP connection. Unlike HTTP (request-response, connection closes after each exchange), WebSocket keeps the connection open for continuous bidirectional data flow. We use it for: (1) Video streaming (/ws/video) — the server pushes annotated video frames (base64-encoded JPEG) plus telemetry data at 15 FPS. Using HTTP, the browser would need to send 15 separate requests per second, each with headers — very wasteful. (2) Control channel (/ws/control) — bidirectional: the browser sends motor commands and mode switches, the server pushes AI decisions and detection alerts back. Backpressure: Each subscriber gets a queue (max 2 frames). If the browser is slow (e.g., processing a frame), the oldest frame is dropped so the user always sees the latest feed, not a delayed one.

Q17: What is the optical flow algorithm and why is it useful?

Dense optical flow (Farneback method) computes the apparent motion of every pixel between two consecutive frames. It uses polynomial expansion to approximate the neighborhood of each pixel, then solves for the displacement vector. Parameters: pyramid scale 0.5, 3 levels, window size 15, 3 iterations. It's useful because: (1) It detects moving obstacles that might not be detected by static methods (e.g., a person walking toward the robot). (2) It provides motion magnitude at each pixel — pixels with flow above 2.0 px/frame are flagged as moving. (3) Combined with other detection methods, it improves overall reliability through multi-sensor fusion.

Q18: How is the ESP32-CAM's MJPEG stream consumed by the backend?

The ESP32 sends an MJPEG stream on port 81 as HTTP multipart content (Content-Type: multipart/x-mixed-replace; boundary=frame). The Python backend uses httpx.AsyncClient to stream this response. It reads chunks and looks for JPEG frame boundaries: start marker 0xFFD8 and end marker 0xFFD9. Each complete JPEG is decoded using cv2.imdecode() into a NumPy array (BGR image). The latest frame is stored under an asyncio lock for thread safety. If the stream fails, it automatically falls back to the laptop webcam. This parsing happens in an async loop to avoid blocking other operations.

Q19: What safety mechanisms does the system have?

Multiple layers of safety: (1) Emergency stop — path planner immediately stops when obstacle < 0.25m in the forward corridor. (2) ESP32 auto-stop — firmware stops motors if no command received for 800ms (failsafe). (3) STOP bypasses voting — no delay in stopping (immediate safety override). (4) Pulse-based control — robot stops between movements to re-analyze the scene. (5) Multi-frame voting — prevents single-frame errors from causing wrong turns. (6) Trajectory feasibility check — rejects paths with cost > 0.9 or clearance < 8cm. (7) Recovery behavior — turns toward clearer side if all paths blocked for 5+ frames. (8) Manual override — human input always takes priority and pauses AI for 5 seconds.

Q20: What is Pydantic and how is it used in this project?

Pydantic is a Python library for data validation using type annotations. It provides: (1) BaseModel — data classes with automatic validation, serialization (JSON), and type coercion. We use it for all data schemas: Detection, TrackedObstacle, LaneStatus, AIDecision, FrameResult, SystemTelemetry. (2) BaseSettings (from pydantic-settings) — loads configuration from environment variables and .env files with type validation. Our Settings class defines all configurable parameters with defaults. Benefits: type safety at runtime, automatic JSON serialization via model_dump_json(), clear data contracts between services, and self-documenting configuration.

Q21: What is the difference between the occupancy grid and the cost map?

The occupancy grid is a raw spatial representation: it shows WHERE obstacles are (binary-ish, 0.0–1.0 occupancy probability). The cost map adds intelligence: it shows how DESIRABLE each cell is for navigation. The cost map builds on the occupancy grid but adds: repulsive potential fields (stay away from obstacle neighborhoods), predictive costs (avoid where obstacles are heading), boundary costs (avoid edges), and attractive costs (prefer forward-center). Think of the occupancy grid as "what's there" and the cost map as "where should I go." The trajectory evaluator scores arcs against the cost map, not the raw occupancy grid.

Q22: How does the system handle the Windows event loop issue with MQTT?

Windows uses ProactorEventLoop by default, which is optimized for Windows I/O but doesn't support add_reader()/add_writer() socket operations needed by MQTT libraries (aiomqtt/paho-mqtt). The fix in run.py: (1) Create a SelectorEventLoop explicitly — this supports all socket operations. (2) Set it as the current event loop. (3) Use uvicorn.Server (not uvicorn.run()) and run it with loop.run_until_complete(). This ensures ALL async code — including MQTT, HTTP, WebSocket — runs in the SelectorEventLoop. On Linux/Mac, the default event loop works fine, so standard asyncio.run() is used.

Q23: What is ByteTrack and why is it important for object tracking?

ByteTrack is a multi-object tracking algorithm that's unique because it uses both high and low confidence detections for association. Traditional trackers only use high-confidence detections, which can lose track of partially occluded objects. ByteTrack works in two steps: (1) First association — matches high-confidence detections to existing tracks using IoU (Intersection over Union). (2) Second association — matches remaining low-confidence detections to unmatched tracks. This recovers objects that might be temporarily obscured. In our system, ByteTrack provides persistent track_id values across frames, which the Kalman filter uses to maintain per-object state estimation.
